System-design in a hurry

Schema evolution: from v1 to v2

Most systems start with a minimal v1. Quality comes from identifying failure modes and evolving safely.

Basic v1 pattern

  • A single table with a mutable status.
  • Few constraints, no append-only history.
  • Works for demos; brittle in production.

Question the design

  1. Can retries create duplicates?
  2. Can we reconstruct who changed status and when?
  3. Can we support tenant-level isolation/compliance?
  4. Can we backfill or roll back safely during migrations?

Improve to v2

  • Add record_status_history for lifecycle transitions.
  • Add idempotency_keys for retry-safe APIs.
  • Add audit_logs for actor/source tracking.
  • Introduce read models for dashboard-heavy workloads.

Safe migration playbook

  1. Schema expand: add nullable/new tables first.
  2. Dual-write from application.
  3. Backfill old records in batches.
  4. Switch reads to new model.
  5. Remove old fields in a later release.
