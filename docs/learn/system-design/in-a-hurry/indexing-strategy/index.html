Indexing strategy: design from queries

Indexes are not decoration. Every index should correspond to a critical query shape.

Workflow

  1. List top read APIs and SQL where clauses/order clauses.
  2. Create composite indexes in left-to-right query order.
  3. Benchmark with representative data.
  4. Drop redundant indexes that add write overhead.

Example

SELECT * FROM orders
WHERE customer_id = ?
ORDER BY created_at DESC
LIMIT 20;

Good index:

CREATE INDEX idx_orders_customer_created
ON orders(customer_id, created_at DESC);

Anti-patterns

  • Indexing every column independently without query mapping.
  • Ignoring sort direction and then relying on filesort.
  • Not re-evaluating indexes as workload changes.

Checklist

  • Do hot reads use index-only or near index-only access?
  • Are writes still within acceptable latency after new indexes?
  • Are stale or duplicate indexes removed?
